# 08 测试计划

## 8.1 测试目标

验证 `bookMS` 在已确认功能范围内的正确性、稳定性与可用性，确保关键业务规则严格生效。

## 8.2 分块交付必附测试清单（强制约定）

### 8.2.1 目的

每完成一个可交付的功能块（例如：认证与会话、图书 CRUD、借阅归还等），**必须同时交付一份可照抄执行的手工验证清单**，便于：

- 在合并代码或进入下一功能前完成回归；
- 使用 Postman、curl 或浏览器在无前端的情况下验证 API；
- 与产品/协作方对齐「什么叫做完」。

### 8.2.2 交付物要求

每个功能块交付时，清单中至少包含：


| 项    | 说明                                          |
| ---- | ------------------------------------------- |
| 前置条件 | 环境、依赖、数据库种子、配置文件等                           |
| 接口列表 | 方法、完整 URL 路径、是否需要 Cookie                    |
| 逐步用例 | 每一步的请求方法、Headers、Body 示例、**预期 HTTP 状态码**    |
| 预期响应 | 关键 JSON 字段（如 `code`、`message`、`data` 中的必填项） |
| 负向用例 | 未登录、无权限、参数错误、业务冲突等                          |
| 可选   | curl 一键命令、Postman Collection 导出说明           |


清单正文可写在：

- 本文件对应小节（如 **8.6 认证模块手工验证清单**），或
- 当单次迭代用例过多时，在 `docs/` 下增加专项补充文档，并在交付说明或本文件 **8.2** 节内注明该文档路径。

### 8.2.3 流程约定

1. 开发完成某功能块 → 编写或更新该块的**完整手工验证清单**。
2. 执行人（默认实现者或指定测试人）**逐项执行并勾选/记录结果**。
3. **全部通过**后，才进入下一功能块的开发或合并主线。

### 8.2.4 清单编写格式建议

- 用例编号：`模块-序号`（如 `AUTH-01`）。
- 每个用例固定包含：**步骤 → 预期状态码 → 预期业务结果（一句话）**。
- 涉及 Session 的接口，注明「需先登录」及 Cookie 携带方式（Postman 自动 Cookie / curl `-c`/`-b`）。

## 8.5 关键测试用例（示例）

### 8.5.1 认证

- 注册成功：新用户可创建账号并登录
- 登录失败：错误密码返回认证失败
- 会话失效：未登录访问受限接口返回 401

### 8.5.2 借阅规则

- 正常借阅：库存充足且未超上限时借阅成功
- 超上限借阅：第 6 本借阅被拒绝
- 逾期限制：用户存在逾期记录时借阅被拒绝
- 归还流程：归还成功后库存 +1，记录状态变更为 RETURNED

### 8.5.3 预约

- 有库存预约：允许创建预约
- 取消预约：状态更新为 CANCELLED

### 8.5.4 报表

- 借阅排行榜：按借阅次数排序正确
- 逾期统计：时间范围过滤正确
- 库存统计：总库存与可借库存一致性正确
- 用户活跃度：借阅次数聚合正确

### 8.5.5 导入导出

- 导入成功：模板正确时写入成功
- 导入失败：字段错误时返回失败行信息
- 导出成功：图书、用户、借阅记录均可导出
